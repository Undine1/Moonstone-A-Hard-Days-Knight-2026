**INTRO AUDIO FIXED — forest CRACKLE killed + LIGHTNING-CRASH and MONK-CHANT SFX now play (2026-06-17, this session). Only moon.c edited (Paula model); g_os-gated by `g_audio_on` (audio gen runs only with `--wav`/SDL); cracktro inert; clean build 0 warnings; default 8000-frame run BYTE-IDENTICAL (ic 84,257,355 / 4 unmapped / pc=03b7c0 — audio changes don't touch the CPU trace).**
  - **ALL THREE were ONE root cause in the Paula DMA-(re)start path: the deferred latch.** When a channel went disabled→enabled (DMACON AUDxEN), `paula_update_dma` only ARMED `first=1` + `acc=0` and deferred the actual ptr/len latch to the next `paula_tick_one`. But with `acc=0` the resampling accumulator needs ~`HOST_RATE/chan_rate` host-samples (≈4-5 at PER≈339-428) to cross `0x10000` before the first tick fires — and during those host-samples the channel kept EMITTING THE PREVIOUS BUFFER'S held `sample` at the NEW (re-triggered) VOLUME. Two symptoms:
    - **(a) FOREST CRACKLE (~fr 2333):** the music driver re-triggers ch2 from the forest melody buffer (loc 0x0ac282/0x0ac13c, vol low) onto a one-shot SFX (loc 0x0c18c4, VOL=0x40). For the ~5 host-samples between the AUD2EN write and the first tick, ch2 still held the FOREST sample (+93) but now scaled by the loud SFX VOL=64 → a single isolated ~5-host-sample spike (R jumped 512→7766 then back to 6). Measured forest-window (fr2000-2800) max sample-to-sample jump **7738 → 1680** (1680 = normal music, no pop). Diagnosed by reversing the mix (left bus≈unchanged, right bus 308→5888 → ch1+ch2; ch2 IRQ-reload at the exact ic) + a temporary `-DSPIKE_DBG` per-channel dump that showed `ch2[s=93 v=64 ... f=1]` (first still armed, old loc) at the spike instant.
    - **(b)+(c) LIGHTNING-CRASH + MONK-CHANT SFX barely audible:** the ritual/lightning beats RAPIDLY re-trigger one-shot SFX channels (chant: sustained ch0/ch1/ch3 loc 0x0aebe4 len9128 + 0x0b60e2 len9326, ~1.1s vocal samples re-fired every ~11 frames; lightning crash: a flurry of short ch0 one-shots loc 0x0bxxxx len~760-1500 vol64 per428 at fr~5140-5900 + ch2 0x0b3466 + ch3 0x0c2308). Each re-trigger reset `acc=0` and held the PRIOR one-shot's decayed/silent tail, so the loud new ONSET was muted/delayed every time → the crash/chant came out weak. **The game DOES program these channels correctly** (confirmed via `--audlog`); our model was eating their attacks.
  - **FIX (moon.c `paula_update_dma`, ~L846): latch IMMEDIATELY on DMA (re)start** — call `paula_latch(ch)` right when a channel transitions disabled→enabled (sets loc/len from the latched LCH/LCL/LEN, `first=0`), then prime the held DAC `sample` from the NEW buffer's first byte + set `wordhalf=1`, then `acc=0` + `paula_recalc_step`. So the very first emitted host-sample is the NEW buffer's data at the new volume — never the previous buffer's sample at the new volume. This matches HW (a DMA restart fetches the first sample word immediately; the DAC doesn't replay the old buffer). One-shot END behaviour is unchanged + correct: when a one-shot writes LEN=1 pointing at a silent word (0x0c18c4 = a zeroed word), it loops on that word = SILENCE (verified the word is 0,0).
  - **VALIDATION (headless WAV, `--frames 7000 --wav`):** forest max-jump 7738→1680 (pop gone); per-frame RMS at the documented flash beats roughly 2-3× (platform-flash fr4520 1677→3393, red-knight flash fr4930 885→3395, lightning-burst fr5400 949→3004; left-bus crash-window fr5140-5440 RMS 598→2245); ritual/chant window (fr2900-4400) RMS ~doubled (fr3050-3200 175→929). Overall intro RMS 684→1565, peak 7766→15810. New WAV's only large global jump is fr5145 (782→10600 and STAYS high = the lightning-crash SFX ATTACK, not a pop; the one-shot samples start with 0,0,0,0 then ramp — clean).
  - **NO REGRESSIONS:** default 8000 byte-identical (ic 84,257,355 / 4 / pc=03b7c0); title-music window (fr7600-8050) max-jump 1210 RMS 570 = clean structured music; combat window (fr9300-9990) max-jump 1364 RMS 752 = clean; both still all-4-channel, non-silent. Cracktro (g_os=0, no audio device) unaffected — `g_audio_on` stays 0 so `paula_generate`/`audio_frame` early-return and `g_paula[]` is never consumed (the default-run byte-identical trace proves the CPU path is untouched even though `paula_update_dma` runs).
  - **DIAGNOSTIC POLISH to `--audlog` (so SFX triggers at later beats aren't buried):** (1) the one-shot silent-loop (LEN<=1) IRQ reload is logged only ONCE per (ch,loc) instead of every word (was flooding ~580k lines); (2) AUDxPER writes are logged only when the value CHANGES (the driver rewrites PER every tick with the same value); (3) audlog line budget 4000→40000.
